Intent resolver

Understand the request in context.

The resolver sees the active capability and the current catalog. It decides whether this is a new capability, an evolution, a presentation change, a one-time data question, or a friendly refusal.

Why it matters Classification stays separate from building, so the same resolved request can be executed once and measured honestly.
The safety model

Reads flow. Writes take turns.

This one rule explains most of Aluna’s data architecture. Reading can stay expressive and concurrent. Every canonical change crosses a small, owned seam.

Reading is free.

Reads run together through a physically read-only SQLite connection. Persistent cross-capability reads declare exactly which incarnations they need.

  • Parameterized SQL and joins are allowed
  • Extra columns stay invisible to old code
  • Ad hoc questions leave no capability behind
→ one owner

Mutation is constrained.

Builds receive bounded tickets. Record changes use short leases. One mutation coordinator ensures that unrelated work never slips into an open transaction.

  • No raw mutation SQL in generated behavior
  • Update and delete are bound to a validated record
  • Builds recheck stale context before starting
!
Evolution preserves. Confirmed deletion destroys.

Normal evolution adds or soft-hides fields. Permanent capability deletion is a separate, zero-AI path with confirmation, dependency checks, a tombstone, and idempotent cleanup across records, snapshots, files, and owned event payloads.

What survives

Aluna remembers in layers.

No single store tells the whole story. Select a layer to see what it owns and why its lifecycle differs from the others.

The registry says what Aluna is now.

One active row per capability holds the authored spec, incarnation, version, and pointer to a complete snapshot. The resolver reads its lean prompt context on every classification.

The authored spec is the source of truth.

Each capability owns a table.

Canonical records live in isolated SQLite tables with no foreign keys. Fields are added or soft-hidden during evolution. Cross-capability relationships happen through reads.

Evolution is additive. Confirmed deletion is explicit.

Every live version is complete.

Each immutable filesystem snapshot contains the exact spec, every Handler, the item renderer, inventory and digests, plus frozen tests only when the behavioral tier is on.

Publish first. Activate the pointer second.

Use becomes evidence.

The Event Log captures full interaction context for the implicit loop. Content-free generation metrics record latency, outcomes, model use, Gate work, and overlap decisions.

Product payloads can be purged. Experiment measurements remain.

Bytes live outside SQLite.

Opaque keys point to an S3-shaped object store. Durable ownership and cleanup keep failed uploads, replacements, record deletion, and capability deletion recoverable.

The AI uses storage. It never builds storage.
The ownership line

Stable mechanics, generated meaning.

Aluna stays coherent because platform code owns the invariants shared by every capability while AI authors the parts that must understand this capability.

The platform owns

  • Shell, navigation, forms, list layout, modal, and accessible item frame
  • Routing, record targeting, mutation coordination, and additive DDL
  • Gate mechanics, publication, activation, recovery, and deletion choreography
  • Read-only query and file-storage infrastructure
SPEC → DERIVED

The AI authors

  • The capability’s complete spec and user-facing intent
  • Five Action Handlers for create, read, update, delete, and search
  • One item renderer for capability-specific record composition
  • Behavioral tests when the experiment’s test tier is on
How it comes together

Eight runnable slices.

The roadmap is vertical, not layered. Every module leaves a behavior a person can run and verify, then deepens the same shared spine.

Module 1

Stand up the runtime spine.

Connect the static shell, SSE transport, dual SQLite connections, storage ports, and the pluggable AI provider. There is no capability logic yet.

Milestone: every dependency is real and wired before generation begins.

If you remember five things

Carry this mental model.

  1. The spec is authored. Code, presentation, and tests are derived.
  2. Reading stays expressive. Every write crosses one constrained seam.
  3. A complete candidate passes the Gate before one atomic activation.
  4. The explicit and implicit loops share one presenter-independent Builder.
  5. Aluna feels like a friendly app, never a tool for steering generated code.
