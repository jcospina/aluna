# Omni-CRUD: High-Level Architecture

> A self-building CRUD runtime. The user states intent — explicitly (prompts) or implicitly (behavior) — and the app evolves to fulfill it. No predefined domain model or capability business logic, no developer in the loop; the platform supplies shared presentation mechanics.
>
> This document describes **what each moving piece is and how they relate**. It is not an implementation plan. Research grounding lives in [RESEARCH.md](RESEARCH.md).

---

## 1. What This Is: a Platform, Not an App

omni-crud is a **platform that builds apps at runtime**. It ships a small, fixed set of things and contains **zero domain logic**:

- the **shell** (the one fixed UI surface),
- the **capability registry** (the source of truth for everything the app has become),
- **generation-time tooling** (the tools the AI uses to build a capability, like a coding agent using its tools),
- the **router** (deterministic dispatch from UI to capability),
- the **capability API** (the UI↔server contract),
- **platform presentation** (shared list, modal, form, field, and item-wrapper behavior),
- **schema/DDL ownership** (the platform manages the database structure).

Everything capability-specific — its CRUD business logic, data shape, presentation intent, and item composition — is **generated by AI at runtime** and persisted. The platform owns only reusable presentation mechanics and never implements a single capability create/read/update/delete. If it did, the entire premise would collapse: the point is that the *app writes itself*. The ownership line is **platform presentation, generated behavior** (ADR-0005).

This applies the [SelfEvolve](https://arxiv.org/abs/2604.16314) pipeline shape —
test generator + code synthesizer + isolated execution + context memory, gated on
executed tests — to the CRUD domain. Execution is isolated from user data through
the scratch database; ADR-0003 deliberately defers a process sandbox for this
single-user local PoC. (The test-generator stage is the **behavioral gate**,
tiered for the latency experiment; see §6.2.)

## 2. Locked-In Decisions

These are settled and form the foundation everything else is built on:

| Decision | Choice | Reason |
|---|---|---|
| **Platform, not app** | omni-crud ships tooling + shell + registry; AI generates all capability logic | The app building itself *is* the thesis |
| **AI generates real code** | Handlers and the item renderer are actual `.ts` files on disk, not interpreted specs | Fixed code produces fixed results; the experiment is whether the model can write capability logic and composition |
| **Not React / Next.js** | Rejected | Too slow to render, too heavy, too much boilerplate for a single-surface PoC |
| **Speed is a first-class constraint** | Optimize for it everywhere | The PoC's thesis includes *how fast* an app can build itself — latency is part of the experiment |
| **Capability registry is the core artifact** | Structured spec, persisted | Validated by MDE research — the spec is the "model", everything else is generated from it |
| **Two intent loops** | Explicit (prompt) + Implicit (behavior) | The core UX research question |
| **Complete event tracking** | Every action logged with context; append-only | Required for implicit intent inference and as the experiment's dataset |
| **Single-user, persisted** | Isolated per user, survives sessions | The app "remembering you" is the value proposition; multi-user adds auth noise without UX insight |
| **Friendly app, never an engineering tool** | All user-facing language is product voice | "Look how easy / friendly / fast" is part of what's being demonstrated |

### The generated artifacts

For each capability, the AI generates the following, all derived from the spec:

1. **Spec** (`schema + ui_intent + behavior`) — the **source of truth**. The only artifact that cannot be reconstructed from something else.
2. **Handler code** — real `.ts` files on disk implementing the capability's logic.
3. **Item renderer** — real generated code for the capability-specific presentation of one record. Platform-owned list, form, modal, and item-wrapper modules compose it into the View.
4. **Behavioral tests** *(tiered)* — executable assertions derived from the spec's `behavior` field, used to gate the build (§6.2). Generated *independently of the handlers* — from the stated intent, not the code — so a passing test is evidence the logic does what was asked, not merely that it agrees with itself. Produced only when the behavioral-gate tier is on; their generation and execution cost is measured so the tier can be weighed against the no-test baseline.

**The spec is authored; handlers, the item renderer, and tests are derived,
version-keyed caches.** Spec evolution regenerates them only when the spec
version bumps, so model nondeterminism cannot silently drift the UI, logic, or
checks. The one separate trigger is an explicit platform artifact-contract
upgrade: it re-derives caches under a new platform-owned contract marker and
flips the pointer atomically without pretending user intent changed. The arrow
still only points spec → {handlers, item renderer, tests}. Platform presentation
is ordinary versioned application code, not a per-capability cache.

### What is deliberately *not* locked in

This document fixes the **architecture, the intended behavior, and the module boundaries**. It is **not** an implementation plan, and it intentionally leaves the rest open — to be decided *inside the module that owns it*, against the constraints that actually exist when it is built:

- **Implementation detail.** Types, classes, function shapes, file layouts, templates, libraries, colors, copy — none of it is specified here. Pinning it down now would be guesswork the build is free to overrule.
- **The implicit-loop UX.** The *backstage* of the implicit loop is defined (see §8); how it *presents to the user* is not. That is flagged again where it lives.

Software is not specified front-to-back in advance: constraints surface, tools change, and better ideas arrive mid-build. What is locked here is the skeleton everything hangs off — not the flesh.

## 3. The Unifying Principle: Mutation Constrained, Reads Free

Nearly every design decision below collapses to a single line:

> **Mutation is constrained and serialized. Reading is free and concurrent.**

This shows up everywhere:

| Axis | Write / mutate | Read |
|---|---|---|
| **Data access** | Constrained capability tool (no raw SQL) | Unconstrained read-only SQL (`SELECT` + joins) |
| **Structure** | Isolated table per capability, no foreign keys | Relationships materialize at query time via joins |
| **Schema lifecycle** | Additive-only — structure is never destroyed | — |
| **Records** | User deletes own records freely (logged) | — |
| **Orchestration** | Builds globally serialized (one at a time) | Reads run concurrent, never queue |

The reasoning: every danger of unconstrained access (corruption, drop-table, integrity drift, races) is a danger of *mutation*. A read mutates nothing. In a single-user, local, BYO-key PoC, the only thing that would normally forbid full read access — security on a deployed multi-tenant system — does not exist. So reads are set free and writes are kept safe.

## 4. Tech Stack

Chosen for minimal overhead, native speed, and the fact that **LLMs generate HTML, SQL, and TypeScript natively and reliably**.

| Layer | Choice | Why |
|---|---|---|
| **Runtime** | Bun | Fastest cold start, native TypeScript, built-in SQLite, imports generated `.ts` directly |
| **Server** | Hono | ~14KB, runs on Bun directly, no framework ceremony, one route file |
| **Transport** | SSE (Server-Sent Events) | Native streaming, 3 lines of client code, perfect for "watch the UI build itself" |
| **Client shell** | HTMX + Alpine.js | HTMX swaps server HTML into the DOM with no build step; Alpine handles local interactivity. Zero framework in the critical path |
| **Persistence** | `bun:sqlite` | Zero-dependency, in-process, microsecond queries. A separate **read-only connection** serves the read/query path |
| **File storage** | `Bun.file` / `Bun.write` (local FS), S3-shaped provider | No separate storage process (same "no dependency tax" logic as choosing SQLite over Postgres). Zero-copy `sendfile` streaming; swap to R2/S3/Garage on deploy by config |
| **AI** | Fast variants of flagship models — Claude Opus (fast mode), top GPT (fast tier), pluggable — behind the **Vercel AI SDK** as the in-process provider spine | Capability quality matters more than per-call cost; latency is part of the thesis. BYO-key keeps the open-sourced demo free. The SDK supplies streaming + structured output + a bounded tool-loop so we don't hand-build a streaming client; provider-agnosticism comes from targeting the Anthropic-/OpenAI-compatible wire shapes (see ADR-0003) |
| **Spec format** | JSON (AG-UI–aligned shape) | Open, structured, diffable. Not invented from scratch |

The shell ships as a single static HTML page with HTMX attributes. Fixed
platform presentation renders its structural surfaces; capability-specific item
markup and handler responses are generated.

### Model strategy

The system targets the **fast variants of each provider's most powerful model**, selected **globally** (one configured model serves the whole run — no per-task routing). This keeps the experiment clean (results attributable to a single model), the config trivial (one key), and removes a routing layer. Comparing models means running the demo twice, not mixing them in one session. Architecturally this implies a **thin, pluggable provider interface** — the orchestrator depends on a `generate(prompt, schema)` contract, not on a specific SDK.

That contract is realized by the **Vercel AI SDK** (in-process, fetch-based, runs on Bun, BYO-key) behind a provider registry keyed by `baseURL`. Because the Anthropic Messages API has become the de-facto wire format for coding models, that registry swaps the global model with one config change across **Claude, GPT, Gemini, and the open Chinese coding models** (Qwen3-Coder, GLM, Kimi, MiniMax, DeepSeek) — all of which expose an Anthropic-compatible endpoint. The SDK owns the streaming client, retries, structured-output validation, and the multi-provider switch; it does **not** own the harness *discipline* (the build pipeline, Diff Engine, layered gate, migration runner), which stays ours because it *is* the thesis. The code-writing step is a loop **bounded to a single build unit** (write → type-check → fix), agentic within a unit but deterministic across them — not a roaming autonomous agent, which would fight the spec→derived-caches discipline (§9.1). Rationale, the rejected autonomous-agent and hosted-API shapes, and the deferred execution sandbox are recorded in ADR-0003.

## 6. The Blocks

### 6.1 Shell — the only fixed UI

A single static HTML page. It never changes after first load. Three parts:

**Prompt Bar** — always visible, free-form text input. It is *context-aware*: it knows which capability (if any) is active, so "add a due date" scopes to the open capability. Submits to the orchestrator; receives streamed HTML back.

**Capability Toolbar** — starts empty (on a fresh user) or rehydrates from the registry on load. Each capability the app has built is an entry. Clicking one loads that capability's View into the content area. This is the only navigation. The server can update it out-of-band (`hx-swap-oob`) so building a capability updates toolbar *and* content in one response.

**Platform presentation** — the reusable, data-free mechanics shared by every
capability: list scaffolding, empty state, accessible item wrapper, create/detail
modal, spec-rendered fields, and safe composition of generated item output. It
may interpret structural spec facts such as field type, required state, detail
ordering, and the collection layout (`ui_intent.collection.layout` — a closed
value selecting how the list container arranges items); it may not implement
capability behavior. Its field rendering
stays centralized so later field types (lists in Module 4 and files in Module 6)
extend one place instead of every generated artifact. Capability-specific item
composition remains generated, but the platform enforces the allowed HTML/class
surface at runtime so record content cannot become executable markup.

**Event Tracker** — a **dumb recorder**. Captures every user action (click, hover, dwell, focus, scroll) with full context — timestamp, active capability, element id/type, on-screen data — batches them, and ships them to the server. It holds **no logic**: no pattern detection, no thresholds. All inference lives server-side. This is the raw material for the implicit loop, captured at full fidelity (it is the experiment's primary dataset).

> The shell is dumb on purpose. It may perform fixed local presentation behavior
> such as opening, prefilling, and focusing the shared modal, but it never infers
> intent, applies capability rules, or mutates canonical state. It renders what
> the server sends and reports what the user does.

### 6.2 Orchestrator — the brain (server-side)

A deterministic **router** fronts everything: the UI only ever calls `/capability/:id/:action`, where `:action` is one of the capability's declared tools (`create`, `read`, `update`, `delete`, `search`). The generated UI never invents routes — it targets this fixed convention, and the router loads and runs the matching generated handler file. **Routing is never an AI concern.**

Behind the router, three cooperating modules:

#### Intent Resolver

The entry point for *both* loops. Receives a typed prompt or a gated batch of behavioral events. Assembles context — the capability registry (every capability's `prompt_context`), the active capability, recent events — and asks the AI to classify intent into a structured object:

```json
{
  "type": "new_capability | extend_capability | ui_change | data_query",
  "confidence": 0.85,
  "target_capability": "images",
  "resolution": "extend",
  "proposed_action": "show metadata in each photo's detail view",
  "user_facing_label": "Adding more detail to your photos",
  "requires_confirmation": true
}
```

Two responsibilities live *here*, not in separate modules:

- **Overlap resolution.** Because the resolver already sees the whole registry, it decides — when a new intent touches an existing capability — whether to **extend** the existing capability or create a **namespaced new one**. (There is no "share a field" option — that would require foreign keys; see §8.) The chosen `resolution` is logged to metrics for the experiment. The user is never asked to reconcile schemas.
- **The friendly label.** The same call emits `user_facing_label` — one warm, product-voice sentence about what's happening. This single phrase threads through the build narration, the confirmation prompt, and the implicit proposal. (No separate generation call for copy.)

#### Capability Builder

Takes a confirmed intent and produces the actual changes through a **global serial build queue** — one build at a time, system-wide (see §8). It works on the **spec first**, then compiles, **atomically**:

1. Generate/update the capability **spec** (`schema + ui_intent + behavior`) — the diffable source of truth.
2. Derive and run an **additive-only migration** (new tables/columns, or soft-hide; never `DROP`/destructive `RENAME`).
3. Regenerate only the **affected units** (see Diff Engine): handler `.ts` files and the capability's single item renderer, written to a path namespaced by spec version and artifact contract. The platform injects the same capability-scoped presentation adapter into every handler, so create, read, and later search results cannot drift. When the behavioral tier is on, each affected handler's **behavioral test** is regenerated alongside it — from the spec's `behavior`, not from the handler — and travels with that unit.
4. **Validate before commit — a layered, fail-closed gate.** Any active rung failing fails the whole build:
   - **Structural** (always): type-check the generated handlers and item renderer and assert they export the required interfaces.
   - **Smoke** (always): run a smoke invocation against the capability API with a synthetic record — proves the handler executes and round-trips.
   - **Behavioral** *(tiered)*: execute the step-3 tests against the capability API, asserting the spec's `behavior` actually holds (e.g. *"title is trimmed and required"*, *"search ranks by recency"*). This is the rung that lifts the gate from *"compiles and runs"* to *"behaves as specified"* — SelfEvolve's TDD checkpoint, adapted to the CRUD domain. On a behavioral failure the Builder may **retry the affected unit** a bounded number of times — feeding the failing assertion back into generation — before declaring the build failed.
   - **Design lint** *(from Module 3, always)*: render the item with synthetic and hostile field values; reject unknown presentation primitives, off-token styling (raw values on the token-owned color/font/type-scale/spacing/border axes — inline `style` is otherwise a permitted, token-disciplined escape hatch), interactive/executable markup, or unsafe field interpolation. Platform-owned item payload and modal hooks are tested as platform code rather than regenerated and re-gated per capability.
5. **Commit = one pointer flip**: for spec evolution, bump the registry version; point the row at the new files, render the committed View, and swap View + toolbar (oob) in a single SSE response. An artifact-contract upgrade changes the contract marker and pointer without bumping the authored spec version.

Throughout, it **narrates progress in product voice** (`user_facing_label`-driven) over SSE — the user watches it build, but nothing is live until commit. On any failure: roll back the migration transaction, leave the previous version live, orphan the new files for GC, and **log the failure to metrics** (failure is data).

**Why the behavioral rung is a tier, not an always-on default.** Generating a test suite and executing it adds a generation pass and an execution pass — directly onto the build latency the PoC exists to measure. Making it a global toggle keeps the no-test path as a clean baseline, so the experiment can *quantify* what behavioral verification costs in wall-clock and how much drift it actually catches, rather than assuming either. This mirrors the model strategy (§4): one knob, set globally for a run, compared by running the demo twice — never mixed within a session.

Every generation writes a metrics record (spec-gen, code-gen, presentation-gen,
test-gen, migration, total wall-clock, per-rung gate outcomes including design
lint, test-run time and any retries, model, tokens, outcome, overlap resolution)
before returning. The row also records the artifact contract. Module 2's
historical `html-gen` measurement is the first presentation-gen shape; Module 8
compares the semantic stage across artifact-contract versions rather than
assuming every version writes `.html` files.

#### Diff Engine

Computes the minimal change between the old and new spec at **unit granularity**
— action handlers and the item renderer. A new field on notes changes the
platform-rendered form/detail surface deterministically from the new spec and
regenerates the create/update handlers that persist it. It regenerates the item
renderer only when `ui_intent` says the item presents that field. Unaffected
handlers and the item renderer stay byte-identical, and their behavioral tests
are neither regenerated nor re-run. When a generated unit changes, the Builder
feeds the AI its previous version plus the spec diff with instructions to apply
the delta and preserve the rest. The Diff Engine then streams the re-rendered
affected surface via SSE with targeted `hx-swap`.

### 6.3 Persistence — partly generated at runtime

Four stores in `bun:sqlite`, plus generated code files and an object store on disk.

#### Capability Registry — the source of truth

One row per capability. The structured spec is canonical; a pointer to the
current versioned generated artifacts lives alongside it. The platform also
records which artifact contract produced that pointer so an application upgrade
can distinguish Module 2's generated views from Module 3's item-renderer shape.

```json
{
  "id": "notes",
  "label": "Notes",
  "version": 3,
  "schema": {
    "fields": [
      { "name": "text",       "type": "string",   "required": true },
      { "name": "tags",       "type": "string[]", "added_in_version": 2 }
    ]
  },
  "ui_intent": {
    "item": "A text-forward card that emphasizes text and treats tags and date as metadata.",
    "collection": { "layout": "feed" },
    "detail": { "shows": ["text", "tags", "created_at"] }
  },
  "behavior": "Title is trimmed and required. Search ranks by recency. Tagging 'urgent' is allowed but not special.",
  "tools": ["create", "read", "update", "delete", "search"],
  "artifact_contract": 2,
  "artifacts_path": "capabilities/notes/v3/contract-2/",
  "prompt_context": "Stores text notes. Users tag and search them."
}
```

`schema` + `ui_intent` + `behavior` are authored by the AI and are what evolves.
`ui_intent` records only capability-specific choices: the item's design
direction, the collection layout (a closed `feed | grid` value the platform list
container reads), and the fields/order shown in detail. Fixed platform choices
such as "detail uses
the shared modal" do not belong in the AI-authored spec. `artifacts_path` points
to the version directory holding handlers, the item renderer, and generated
tests. `artifact_contract` is platform-owned compatibility metadata, not AI
intent. The registry row stays lean (spec + version + contract + pointer), which
matters because the Intent Resolver scans every row on every classification.
`prompt_context` is what the Intent Resolver reads to understand existing
capabilities.

Field types include a `file` / `file[]` type. A file field stores only a **reference** (storage key + mime + size + original name) in the data table — never the bytes (see §6.3 Object Store and §7 Files). So a `photos` capability is just a normal capability whose schema has a `file`-typed field.

The platform form/detail renderer is exhaustive over the committed field-type
vocabulary. Module 4 extends that one renderer when list types arrive; Module 6
extends it again for file controls and file detail presentation. Unknown types
fail closed rather than falling back to an arbitrary text field.

#### Event Log — append-only

Every user action, with full before/after context — not just the change, the *situation*. Queried by the Intent Resolver to power the implicit loop and to retroactively explain intent. Append-only by design; the experiment's primary dataset.

#### Data Tables — additive-only, generated DDL

The user's actual content. One **isolated** table per capability — **no foreign keys** (§8). Created and migrated at runtime by the Builder, **additive-only**: structure is never destroyed, only added or soft-hidden. JSON columns serve as the escape hatch for fields that don't warrant a real column yet.

#### Generation Metrics — the experiment's measurements

One row per generation: timing breakdown, artifact contract, model, token counts,
success/failure, and the overlap resolution chosen. This store is *why the PoC
exists* — latency and capability conclusions come from querying it, not guessing.
Distinct from the Event Log (what the *user* did); this records what the *system*
did to build itself.

#### Object Store — user files on disk

A platform-provided file store, **not** something the AI builds — storage is infrastructure, same as the router and the data tool. The default provider is the **local filesystem** via `Bun.file` / `Bun.write` (zero new process, zero-copy `sendfile` streaming), addressed by opaque key under `storage/<key>`. The tool exposes an **S3-shaped contract** (`put` / `get` / `delete` / `url`), so swapping to R2 / S3 / Garage on deploy is a config change, mirroring the pluggable model provider. Bytes live here; the *reference* lives as a `file`-typed field in the capability's data table.

## 7. Data Access Model

Direct application of the unifying principle (§3).

### Writes — constrained

Generated handlers persist canonical state through a **constrained capability tool** (row-level `insert` / `update` / `delete` keyed by capability). Handlers **never** do their own DDL or raw mutation SQL — the platform owns schema. *Internally*, a handler is free to do whatever else it needs (compute, `fetch` an external API to enrich a record); that's out of scope for the experiment. The line is: **canonical state goes through the tool; incidental I/O is the handler's business.**

### Reads — free

All reads — both a capability's generated `read`/`search` actions *and* ad-hoc `data_query` — run **unconstrained read-only SQL** (`SELECT` + joins, including across capability tables). Safety is enforced **deterministically** by a **read-only SQLite connection** (`SQLITE_OPEN_READONLY` / authorizer) — a write is physically impossible regardless of what SQL the model emits. The model is given the registry's schema as a catalog so it writes correct SQL. A cheap LLM classifier may *route/reject* obvious non-queries early ("delete everything" → friendly refusal) but is **never** the safety boundary.

### Files — a platform tool, same split

File storage is **platform tooling** — the AI never builds a storage system (out of scope, brittle, pointless). It calls a provided **S3-shaped tool**, default-backed by the local filesystem (`Bun.file` / `Bun.write`), swappable to R2 / S3 / Garage by config.

- **Upload = write = constrained.** Uploads arrive through the generic router (`/capability/:id/create`, multipart); the generated handler calls `files.put(bytes, { mime, name }) -> { key }` and stores the returned key (+ mime, size, name) in the record via the data tool. The AI writes *logic*, never a storage system.
- **Serve = read = free + infrastructure.** A platform-owned route `/files/:key` streams bytes via `Bun.file`. Generated HTML simply references `/files/<key>` (e.g. `<img src>`) — the AI never builds file serving, exactly as it never builds routing.
- **Lifecycle follows the record.** Deleting a record deletes its file (user-driven, logged) — consistent with §3 (records freely deleted; structure never destroyed).

### `data_query` — the ephemeral exception

`data_query` ("how many photos tagged sunset?", "notes from last week") builds nothing. The AI translates NL → read-only SQL, runs it, and renders results through a **platform-owned generic auto-table** (presentational only, so it doesn't violate "no platform business logic"), with a defensive `LIMIT` + timeout. It is **never persisted** — no registry entry, no toolbar tab, no version, no cache. It is the deliberate exception to "everything is cached," because a data read is keyed on *current data*, not on a spec version. Scope follows the context-aware prompt bar.

## 8. The Two Loops

### Loop 1 — Explicit (prompt → capability)

```
User types in Prompt Bar
        │
        ▼
Intent Resolver  ── classify (new/extend/ui/query) + overlap resolve + friendly label
        │
        ▼
Capability Builder  (enters the global serial build queue)
        ├─ write/update spec       (registry, source of truth)
        ├─ additive migration      (data tables)
        ├─ regenerate changed units (handler + item renderer + test, version-namespaced)
        ├─ validate (typecheck + smoke + behavioral tests + design lint)
        └─ commit (pointer flip)   ── narrate progress in product voice throughout
        │
        ▼
Diff Engine  ── stream only changed fragments via SSE
        │
        ▼
Shell  ── HTMX swaps content area + toolbar (oob) in one response
```

The user watches the UI assemble itself — narrated in friendly language, never internals — and only sees committed, validated results.

### Loop 2 — Implicit (behavior → proposal → capability)

```
Event Tracker (dumb) batches actions → server
        │
        ▼
Server-side gate  ── cheap deterministic heuristic; only trips on a real pattern
        │  (no LLM call until it trips)
        ▼
Intent Resolver  ── async, off the interaction path: reads event batch + context
        │
        ├─ confidence < threshold ──▶ log only, back off (raise bar for this pattern)
        │
        └─ confidence ≥ threshold ──▶ friendly proposal via SSE
                                              │
                                  ┌───────────┴───────────┐
                                  ▼                       ▼
                            user confirms           user ignores
                                  │                       │
                                  ▼                       ▼
                          enter Loop 1            log dismissal,
                          (build it)              back off pattern
```

The gate lives **server-side** so its thresholds — the experiment's main tuning knob — sit next to the event-log dataset and can be tuned without redeploying the shell. Inference runs **async** and never blocks interaction. The implicit loop **never silently changes the app** — it proposes, and confirmation hands off to Loop 1.

> **The implicit UX is not yet defined — deliberately.** Everything above is the loop's *backstage*: event capture, the server-side gate, async inference, and the confirm → hand-off contract. How any of it **surfaces to the user is an open design question**, owned by the module that builds the implicit loop — whether a proposal interrupts or waits quietly in the background, where and how it appears on screen, whether there is a persistent assistant presence or a fleeting notice, its tone and its timing. The only fixed commitment is the contract itself: the implicit loop **never changes the app without an explicit confirmation**, and confirmation hands off to Loop 1. Everything about *how that is shown* is free to change.

### Overlap resolution: extend or namespace

When a new intent overlaps an existing capability, the Intent Resolver (which sees the whole registry) decides silently: **extend** the existing capability, or create a **namespaced new one**. There is **no shared-field / foreign-key option** — capabilities are isolated tables, and any cross-capability need is met by read-only SQL joins at query time, not by stored relationships. The user never reconciles data models. The chosen resolution is logged for the experiment.

### Concurrency

**Builds are globally serialized** — one at a time, system-wide — via a single queue both loops enqueue into. This eliminates every version race at zero real cost (single user) and keeps the "watch it build" narration coherent (one story at a time). **Reads run fully concurrent** and never queue. A pending implicit proposal simply waits (non-blocking) for confirmation, then joins the queue.

## 9. Operating Principles

1. **Spec is the source of truth; handlers, the item renderer, and tests always follow.** All are version-keyed derived caches, regenerated only on a spec bump, never otherwise — so AI nondeterminism can't cause drift. The arrow only ever points spec → {handlers, item renderer, tests}. A platform artifact-contract upgrade may re-derive caches without changing user intent, but it uses a separate contract marker and atomic pointer flip; it never masquerades as a spec evolution.

2. **Mutation constrained, reads free.** Writes go through the constrained tool against platform-owned, additive-only schema; reads run unconstrained read-only SQL behind a read-only connection. (See §3.)

3. **Structure is never destroyed.** Migrations are additive-only — add or soft-hide, never `DROP`. The AI drives schema changes, so this makes the system *structurally incapable* of AI-caused data loss. Confirmation is therefore reserved for exactly two things: **deleting a whole capability** and **every implicit-loop proposal**. Field-level evolution is silent and instant.

4. **Merge conflicts are resolved silently — never surfaced.** The Intent Resolver decides extend-vs-namespace automatically and proceeds. Getting it wrong sometimes is an acceptable, measurable PoC outcome; bugging the user is not.

5. **Builds are atomic and validated through a layered, fail-closed gate.** Nothing goes live until it clears every active rung — type-check, interface assertion, smoke run, Module 3's design lint, and (when the tier is on) behavioral tests generated from the spec's `behavior` — then commits via a single pointer flip. The behavioral rung is what makes "validated" mean *behaves as specified*, not merely *compiles and runs*; it is a global toggle so its latency cost stays measurable against the no-test baseline (§6.2). A failed build never bumps the version. Underneath, broken code is never shipped — even though the surface stays friendly.

6. **Generation is measured, and slowness is expected.** Latency is data, not a defect. Every generation writes a metrics record; conclusions come from querying the dataset.

7. **Friendly app, never an engineering tool.** No internals ever surface to the user — no "handler," "migration," "compile," "spec." Narration, proposals, confirmations, and errors all speak in product voice. "Look how easy / friendly / fast" is part of the thesis.
