Aluna. Module 4 field guide

Technical field guide

How a capability changes without losing itself.

Module 4 lets people edit, search, extend, and permanently delete what Aluna has built for them. It also protects saved data when a capability changes and keeps the previous version working if a new version fails. This guide explains how, one idea at a time.

01 / 12

What changes in Module 4.

Earlier modules taught Aluna to build and display a capability. Module 4 makes that capability safe to edit, extend, recover, and permanently delete.

At the end of M3

Create and view records

A capability can save new records and show them in its own visual style. Changing what it tracks still requires rebuilding it from scratch.

→

At the end of M4

Use and safely change records

A capability can create, view, edit, search, and delete records. It can also gain new fields or presentation changes without losing existing data.

The important change is larger than adding Edit and Delete buttons. Aluna must know which version is currently live, keep older completed versions intact, and move to a new version only after that version has passed every check.

Capability data lives in SQLite, while generated files live on disk. Those two places cannot be updated as one all-or-nothing operation. Module 4 therefore sets a safe order: prepare and verify the files first, then switch the database to the new version. If anything fails early, the old version remains live.

What the user experiences

“Add a due date to my notes.”

One prompt, a clear progress explanation, preserved notes, and then a refreshed Notes view with the new field.

What Aluna does behind the scenes

Prepare, check, and switch versions.

Understand the request, prepare a complete new version, determine what must be rebuilt, test everything together, save the version, and only then make it live.

!

M4 does not turn Aluna into a coding platform.

Users describe outcomes. They do not select types, edit migrations, tune CSS, or steer repair steps. The developer panel remains read-only. AI authors capability meaning. Fixed Aluna rules protect data, version history, and the safe order of each change.

02 / 12

One capability needs three identifiers.

“Notes” tells us what the capability means, but Aluna also needs to distinguish one lifetime of Notes from another and one saved version from the next.

id = notes

Capability name

The stable meaning. It remains notes while Notes changes.

incarnation = 4a80…

Lifetime identifier

Unique to this Notes. Delete and later rebuild Notes, and this value changes.

version = 3

Version number

Counts completed versions within this lifetime: v1, v2, v3, and so on.

Why each lifetime gets a new identifier

The runtime can remember generated code by its file path. If a deleted Notes capability later reused the same path, Aluna might accidentally load old code or let unfinished cleanup touch the new Notes. A new lifetime identifier gives the rebuilt capability a new path and keeps old cleanup separate. 

capabilities/
└── notes/
    └── 4a80b52d…/          ← incarnation
        ├── v1/
        ├── v2/
        └── v3/             ← active registry pointer
            ├── spec.json
            ├── snapshot.json
            ├── create.ts
            ├── read.ts
            ├── update.ts
            ├── delete.ts
            ├── search.ts
            ├── item.ts
            └── behavioral tests, only when behavior testing is enabled
i

Every saved version is complete.

Each version folder contains the complete capability description and every generated file needed to run it. Aluna never creates a live version by mixing files from several versions.

The manifest is a receipt for what was built

snapshot.json lists every file in the version and records a digital fingerprint for its contents. For each generated file, it also records which exact versions of other capabilities and which approved context were used to create it. If a file is safely copied forward, that original evidence is copied too.

This information helps developers verify and investigate a build. It does not decide which version runs or automatically force another capability to rebuild. The manifest lists itself, but it cannot fingerprint itself because that fingerprint would change the file it is trying to fingerprint.

In plain language

Think of a snapshot as a sealed release folder, the manifest as its packing list with digital fingerprints, and the registry as the bookmark that tells Aluna which sealed folder to run. The packing list proves what was built; the bookmark decides what is live.

03 / 12

Five consistent ways to work with records.

Each capability can behave differently, but every capability uses the same five Actions and the same safe request paths. Generated code never invents a new route or chooses which record it is allowed to change.

First, the acronym

CRUD means create, read, update, and delete: the four basic ways to manage saved records. M4 completes that set and treats search as a fifth fixed Action.

Action HTTP What generated code decides What Aluna always protects
create POST Interpret the submitted values and apply capability-specific behavior. Only visible, allowed fields are saved, and every required value is present.
read GET Choose and order the capability’s records. The database connection cannot write or delete data.
update POST Apply capability-specific rules to the submitted changes. Aluna has already chosen and checked the exact record that may change.
delete POST Apply capability-specific deletion rules and wording. The exact record is fixed, and a missing record always fails consistently.
search GET May reorder the records that already match the shared search rules. Text and list matching, character normalization, exclusions, and default order stay consistent across capabilities.

Reading data

Generated code may choose and combine records.

A capability may read its own records and records from other capabilities it has explicitly named as dependencies.

  1. The database connection is unable to write.
  2. Every result column has a declared name and type.
  3. Reading another capability requires naming its exact lifetime.
  4. Record lists return unique record identifiers in the chosen order.

Changing data

Generated code receives only the change it may make.

Aluna gives generated code a limited tool for one capability. For Edit and Delete, that tool is already tied to one checked record.

  1. Generated code cannot choose another table, capability, or record.
  2. Unknown, hidden, and Aluna-owned fields are rejected.
  3. Aluna combines changes and checks required values.
  4. One shared coordinator prevents writes from overlapping unsafely.
≠

These safeguards prevent mistakes, not hostile code.

Generated code still runs inside the Aluna server. Limited tools and code checks block ordinary generation mistakes and known unsafe patterns. They are not a security boundary designed to contain deliberately malicious code.

Every capability follows the same minimum search rules

Search behavior What the user can expect
Terms A search is split into words wherever Unicode defines whitespace. Every word must match somewhere, even when different words match different fields or list entries.
Normalization Aluna cleans up both the query and stored text in the same way and compares them without case differences. This lets visually equivalent character forms match consistently. The exact JavaScript rule is normalize("NFKC").toLocaleLowerCase("und") for the exact rule.
Eligible values Search checks visible text fields and every entry in visible text lists. It does not search hidden fields, Aluna-owned data, or non-text values.
Literal input Percent signs, underscores, quotes, and other database punctuation are searched as ordinary text. They never become database commands or wildcard instructions.
Empty query An empty search shows the same records and order as the normal list. Clear returns directly to that normal list.
Set and order Results never repeat. By default, newer records appear first, with record id as a consistent tie-breaker. A capability may reorder those same matches, but it may not add or remove matches.
After Create, Edit, or Delete After Create, Edit, or Delete, Aluna repeats the current search. If there is no search, it reloads the normal list. The whole records area refreshes together.

Why generated reads return record identifiers first

Generated Read and Search code chooses which records match and in what order, then returns only their unique identifiers. Aluna immediately loads the complete current record for each identifier. This prevents older copied code from leaving out a field that was added in a newer version.

The complete database row stays inside trusted Aluna code. Generated behavior sees only the visible fields allowed for its Action, and the item display sees only the fields it was told to show. Hidden and internal values never need to reach the browser.

04 / 12

Fields can change without erasing old data.

A field can be added, hidden, or shown again. Aluna keeps its original internal name, data type, database column, and saved values throughout those changes.

See what happens when due_date changes

Field definition in the proposed version

What Aluna does

    Hiding a field does not erase it

    A hidden field remains in the capability definition and is marked inactive. Its database column and saved values remain, but new forms, details, search, generated behavior, and required-value checks ignore it. Showing the field again reveals its original saved values.

    The internal field name and data type therefore never change. User-facing wording can change by updating the label. If the field’s meaning truly changes, Aluna should create a new field so old values keep their original meaning.

    A new required field does not break old records

    Database columns created by Module 4 are allowed to be empty. Aluna checks required values when a record is created or edited. An old record may therefore remain readable without a newly added required value, but the next edit cannot be saved until that value is supplied.

    Why this matters

    Nullable storage means an older row may temporarily have no value in a newly added column. Logically required means Aluna will not accept the next create or update unless the completed record has that value. This preserves old data without weakening the new rule.

    01

    Remember what was submitted

    The form sends an internal __aluna_present marker for every visible field, even when its value is empty.

    02

    Check the request

    Aluna rejects duplicate single values or invalid internal markers before generated code sees the input.

    03

    Fix the record target

    __aluna_record_id identifies the one record being edited. Generated code cannot replace that target.

    04

    Keep or clear intentionally

    A field missing from the submitted form keeps its old value. A submitted empty field explicitly clears its old value.

    05

    Check the completed record

    Aluna combines old and new values, checks every required field, and then saves the complete result.

    []

    M4 supports lists of text values.

    A field such as Tags can hold several text entries in order. Empty entries are removed, a comma remains part of the value the user typed, and the list is stored as JSON. Lists of numbers, dates, or other types are intentionally deferred.

    P

    Internal Aluna names cannot become user fields.

    User field names may not start with __aluna_ because Aluna reserves that prefix for safe request handling. The built-in created_at value may be displayed, but it is not a user-editable or searchable field.

    05 / 12

    How one request becomes a safe change.

    The user asks for an outcome. Aluna first understands the request, then waits for a safe turn, prepares and tests a complete new version, makes it live, and finally refreshes the screen.

    0

    No meaningful change means no new version.

    If the checked proposal means the same thing as the current capability, Aluna records “no change” and stops. It does not alter the database, generate code, copy files, create a version, or refresh the screen as if something changed.

    No-op, simply

    When a checked request leads to the same meaning and behavior the capability already has, engineers call it a semantic no-op. Aluna records the outcome but does not create a cosmetic new version.

    06 / 12

    Aluna works out what must be rebuilt.

    Aluna lists every meaningful difference between the current and proposed versions. Fixed rules then decide which data, generated behavior, tests, and presentation must change.

    Choose what changed

    Select one or more changes. The right side combines all the work they require. One change can never cancel work required by another.

    Changes

    Work Aluna must do

    Data and shared interface
      Generated behavior
        Behavior checks

          Facts become effects

          A change fact says what is different, such as “a date field was added.” An effect says what that difference requires, such as “add a database column and regenerate the item renderer.” The Diff Engine is the tested rulebook that maps every admitted fact to all of its required effects.

          Aluna copies old code only when it is known to remain valid

          Fixed rules must explicitly show that a generated file is unaffected before Aluna copies it into the new version. If the file must be regenerated, Aluna may show its old source to the model only after checking that it uses fields, dependencies, and permissions that still exist. Otherwise, generation starts without that old source.

          Copying a known-compatible file is different from using old code as an example for new generation. Old example code might mention a field that is now hidden or a dependency that is no longer allowed.

          ?

          Aluna stops when it encounters a kind of change it does not understand.

          It never guesses that an unfamiliar difference is harmless. Developers must first add and test a rule explaining what work that difference requires.

          07 / 12

          The complete version must pass every check.

          Aluna may regenerate only a few affected files, but it always tests the assembled version as one complete capability before making it live. This final set of checks is called the Gate.

          01

          Files and types

          Confirm that every required file exists, all code types are valid, and the pieces connect through the expected interfaces.

          02

          Five basic Actions

          Create, read, partially edit, search, and delete temporary test records in a separate test database.

          03

          Difficult search cases

          Try accented and equivalent characters, text lists, punctuation, several search words, hidden fields, full records, and stable ordering.

          04

          Expected behavior

          When behavior testing is enabled, run every new expectation and every existing expectation affected by changed code.

          05

          Safe presentation

          When the item display changes, render difficult and potentially unsafe values and confirm that the result follows Aluna’s interface rules.

          Expected behavior comes from the request, not from the generated code

          Aluna creates behavior tests from what the capability is supposed to do, which errors it promises, the fields each Action may use, and the other capabilities it may read. It does not inspect generated code to decide what counts as correct. The expected result is fixed before any repair begins.

          If a test fails, Aluna may repair the generated Action code and then rerun the exact same test. When it is unclear which file caused the failure, Aluna repairs or retests a wider set. It never changes the expectation merely to make the failure disappear.

          Why freeze tests?

          The expected behavior is written down before repair starts, then held still. A repair must satisfy that same expectation; it cannot make a failing test easier. This separates “what should happen” from “how the generated code makes it happen.”

          Tests in current version Tests enabled for new version Expected behavior or code What happens
          disabled disabled any change Do not create or run behavior tests.
          disabled enabled any change Create expectations for the proposed version, fix them in place, and run them.
          enabled enabled unchanged and its Action code is unchanged Copy the existing tests unchanged. They do not need to run again.
          enabled enabled unchanged but its Action code changed Copy the existing tests and rerun the affected tests, or all tests if unsure.
          enabled enabled changed Create new expectations, fix them in place, and run them.
          enabled disabled any change Do not include behavior tests in the new version.
          T

          Changing the test setting alone does not create a new version.

          The new setting takes effect the next time the capability itself meaningfully changes. Aluna does not create an otherwise identical version only to record that behavior testing was turned on or off.

          08 / 12

          Saving a version is different from making it live.

          Aluna first saves and verifies a complete version folder. It then updates the database to make that version active. Keeping those moments separate makes failures recoverable.

          The key distinction

          Publication safely places a verified version on disk. Activation changes the database record that says which version Aluna should run. Publishing first means a crash can leave an unused version on disk, but it cannot leave the live capability pointing at half-written files.

          Files on disk

          Store complete versions that never change.

          Aluna builds in a unique temporary folder, checks the file list and digital fingerprints, then moves the complete folder into its final version path without overwriting an existing version.

          SQLite

          Records which version is live.

          One database transaction adds any needed columns, confirms that the expected previous version is still active, points to the new version, and records a successful activation.

          Prepare

          Write every proposed file into a new temporary folder.

          Verify

          Run all checks and confirm the exact file list and fingerprints.

          Publish

          Move the complete folder into its final version path in one step.

          Make live

          Commit the database change that makes the new version active.

          Refresh

          Send the browser one complete updated capability view.

          Failure before the version becomes live

          The previous version remains active. Aluna records any temporary or completed but unused folder so recovery can safely clean it later.

          Failure after the version becomes live

          The new version stays active. If the browser missed the final refresh or disconnected, reloading reads the database and shows the correct new version.

          Older completed versions remain part of history

          If v3 is live, v1 and v2 are still valid completed versions. Recovery may remove temporary folders and versions proven never to have been active, but it never assumes that an older completed version is disposable. If completed history is missing or damaged, Aluna stops instead of guessing.

          The screen follows the version recorded in the database

          When a new version becomes active, Aluna sends one complete refreshed view. When nothing changed, the request became outdated, the user cancelled, or the build failed, Aluna reloads the current view, clears search and open detail state, and finishes. Rebuilding fewer files never means partially refreshing the screen.

          09 / 12

          Writes take turns so they cannot interfere.

          One coordinator controls every operation that changes shared database state. Builds wait in order, shorter writes take brief turns, and reads continue without joining the queue.

          A controlled queue

          A build first receives a place in line, called a ticket. When its turn arrives, it receives temporary proof that it may write, called a lease. Only one owner receives that proof at a time, so unrelated work cannot accidentally share one database transaction.

          A moment in the queue

          Change Noteswriting now
          Build Recipesnext in line
          Edit note #17try again later
          Save build measurementwaiting
          Read Photosruns now

          Who waits and who can continue

          build
          Waits in arrival order, then owns the write turn while building.
          record write
          Takes a short turn only when no build is already waiting ahead of it.
          platform write
          Waits behind build work, then takes a short write turn.
          capability delete
          Does not wait. If any write owns or awaits a turn, deletion is refused.
          read / search
          Does not enter the write queue and can run at the same time.

          Why Aluna checks the request again after it waits

          Aluna understands a request before placing it in the queue. While it waits, another build may change the target capability or build something new. When the request reaches the front, Aluna confirms that the exact capabilities and versions it originally saw are still current. If not, the request is marked outdated and stops before any AI generation begins.

          Only after this second check does Aluna record that a real build has started and preserve the exact dependency information used for generation. The build record therefore describes work that genuinely received permission to run.

          R

          The build process is separate from how progress is shown.

          The Builder receives a request Aluna has already understood and reports what stage it reaches. Today, the prompt experience turns those updates into progress text. A later module can reuse the same build process for a confirmed suggestion without changing the safety rules inside the build.

          10 / 12

          Permanent deletion follows its own safe process.

          Changing a capability preserves its history and data. Confirmed deletion removes them permanently. Aluna must finish the irreversible database change first and be able to retry slower file cleanup afterward.

          A durable cleanup receipt

          After the permanent database deletion, Aluna keeps a cleanup record called a tombstone. It says the capability is already gone and lists what still needs removal. Each cleanup step is idempotent, meaning it is safe to retry after a crash.

          01

          Confirm plainly

          Name the capability and every category of data that will be lost.

          02

          Check whether deletion is safe now

          Do not wait in the write queue. Confirm the target still exists and refuse if another capability currently depends on it.

          03

          Let current readers finish

          Block new reads and wait for every operation already using the capability.

          04

          List everything to remove

          While the database table still exists, collect its generated files and every capability-owned resource or Event Log payload.

          05 · point of no return

          Make deletion permanent

          Mark the capability unavailable, remove installed payloads, and delete its database table in one committed change.

          06

          Finish cleanup, even after a restart

          Delete the remaining files and resources. Retry safely after a failure, then remove the cleanup record.

          A capability cannot be deleted while another one relies on it

          Suppose Recipes reads information from Notes. Deleting Notes would break Recipes, so Aluna names Recipes and refuses the deletion. Hiding one Notes field does not cause this problem because its stored column remains. Deleting the entire Notes table would remove data that Recipes still expects.

          Deletion waits for operations already reading the capability

          Every operation that uses a capability receives temporary permission to read that specific lifetime. Work that needs several capabilities receives all permissions together or none. Deletion blocks new permissions and waits for current readers to finish. If deletion fails before the permanent database change, reads can reopen. If it fails afterward, the capability remains deleted.

          A cleanup record lets deletion resume after a failure

          After the permanent database change, the capability is gone even if a generated file could not be deleted. A durable cleanup record lists what remains and temporarily reserves the capability name. Startup recovery retries the list. The same capability name can be built again only after cleanup finishes, and the rebuilt capability gets a new lifetime identifier.

          Deletion removes records, the live registry entry, the complete version history, owned resources, and capability-owned Event Log content. Anonymous measurements about past generation remain because they contain no retained user content.

          Before the permanent database change

          The capability still exists.

          If deletion fails, reads reopen and the capability remains visible and usable.

          After the permanent database change

          The capability is gone.

          File cleanup may continue or retry, but the capability, toolbar entry, and records cannot return.

          12 / 12

          Follow one change from request to result.

          This example shows how Aluna safely adds a due date to an existing Notes capability while preserving every saved note.

          Existing Notes v1 is active

          “Add a due date to my notes and make it stand out in the list.”
          01

          Understand the request

          Aluna decides that the user wants to change Notes and remembers the exact Notes lifetime and version it examined.

          02

          Wait for a safe turn

          The request waits behind earlier builds. When its turn arrives, Aluna checks that Notes has not changed and then records that the build started.

          03

          Propose the complete next version

          The AI returns a complete Notes description containing every existing field, the new due_date field, and instructions to show it in the list.

          04

          Check the proposal

          Aluna confirms that existing fields kept their names and types, the new field is valid, and every behavior, dependency, and presentation reference is safe.

          05

          Work out what must change

          Aluna determines that Notes needs a new optional database column and updated Create, Edit, Search, and item-display behavior. It combines all required work.

          06

          Define success before generating code

          Aluna creates the relevant behavior expectations from the proposed Notes description and fixes them in place before generating or repairing code.

          07

          Build and test a complete version

          Affected files are regenerated, known-safe files are copied, and the complete proposed version passes file, Action, search, behavior, and presentation checks.

          08

          Save the verified version

          Aluna writes the complete description and file receipt, verifies the folder, and moves it into the final v2 location in one step. It is saved but not live yet.

          09

          Make v2 live

          One database transaction adds the optional due-date column, confirms that v1 is still active, switches Notes to v2, and records a successful activation.

          10

          Show the result

          Only after v2 is live does Aluna refresh the complete Notes view. Existing notes remain visible, with an empty due date until the user edits them.

          The whole idea in one paragraph.

          The AI proposes a complete next version. Aluna determines what changed, limits what generated code may do, builds and tests the entire version, saves it before making it live, and handles permanent deletion through a separate recoverable process. The old version remains live until one successful database change activates the new one.

          REFERENCE

          Glossary.

          Short, plain-language definitions for the engineering terms used throughout this guide. Each definition describes what the term means specifically in Aluna.

          Action
          One fixed operation a capability can perform: create, read, update, delete, or search.
          active version
          The completed version Aluna is currently using. Older completed versions remain in history but are not shown to the user.
          additive evolution
          A way of changing data safely by adding storage and preserving old fields and values instead of destructively rewriting them.
          atomic
          An operation that becomes visible all at once or not at all. No observer should see a half-completed state.
          Builder
          The process that takes a request Aluna already understands, prepares and tests a complete new version, saves it, and makes it live.
          candidate spec
          The AI’s complete proposal for the next capability version. It describes every field, behavior, and presentation choice, including the parts that did not change.
          capability
          One thing Aluna has built for the user to keep track of, such as Notes, Recipes, or Photos.
          canonical
          The one standard form Aluna treats as the source of truth. Other inputs are converted to or checked against this form.
          CAS
          Compare-and-swap. Make the new version live only if the exact previous version the build expected is still live. Otherwise stop because the capability changed while the build waited.
          change fact / effect
          A change fact says what is different, such as “a field was added.” An effect says what work that difference requires, such as “add a column and update Create.”
          CRUD
          Create, read, update, and delete. The four basic ways to manage saved records. Aluna also exposes search as a fifth Action.
          DDL
          Data Definition Language. SQL that changes table structure. M4 evolution uses additive DDL, such as adding a nullable column.
          deterministic
          Governed by fixed platform rules, so the same admitted input produces the same decision instead of relying on an AI judgment call.
          digest
          A digital fingerprint calculated from content. If the content changes, its digest changes, which lets Aluna detect missing or altered files.
          Diff Engine
          The fixed set of rules that compares the current and proposed capability, lists every meaningful difference, and decides all work those differences require.
          frozen behavioral test
          An expected result written from the requested behavior and fixed before code repair begins. Repair must meet the expectation and cannot make it easier.
          FIFO
          First in, first out. Waiting builds receive turns in arrival order, within a bounded queue.
          Gate
          The complete set of checks a proposed version must pass before it can become live: files and types, all five Actions, difficult search cases, expected behavior, and safe presentation.
          Handler
          The generated code behind one Action. It receives already-checked input and only the limited reading or writing tools that Action needs.
          idempotent
          Safe to retry. Running the same cleanup step again reaches the same result without duplicating damage or corrupting state.
          incarnation
          A unique identifier for one lifetime of a capability. Normal changes keep it. Permanent deletion followed by rebuilding the capability creates a new one.
          manifest
          A version folder’s receipt. snapshot.json lists every file, its digital fingerprint, whether behavior testing is enabled, and evidence about how each generated file was created.
          mutation coordinator
          The part of Aluna that gives database writes safe turns. Builds wait in arrival order, shorter writes take brief turns, and reads do not enter the queue.
          nullable / logically required
          Nullable storage allows an older row to lack a new value. Logical requiredness prevents a create or update from saving a completed row without that value.
          positive proof
          A fixed rule explicitly showing that a generated file is unaffected by the change and may be copied. The absence of a rule is not enough.
          projection
          A deliberately limited set of fields containing only what a particular Action or generated file is allowed to see.
          provenance
          Evidence showing which versions of other capabilities and which approved information were used to create a generated file.
          publication / activation
          Publication saves a checked, complete version folder on disk. Activation updates the database so Aluna starts using that version.
          read token
          Temporary permission for an operation to use one capability lifetime. Deletion blocks new permissions and waits for existing readers to finish.
          registry
          The database directory of capabilities. For each one, it records the current lifetime, live version, complete description, and version-folder path.
          semantic no-op
          A request whose complete validated result means the same thing as the current capability, so no new version is created.
          snapshot
          One complete version folder containing the capability description, all generated files, the file receipt, digital fingerprints, and any behavior tests. After the version is saved, these contents never change.
          SSE
          Server-Sent Events. The one-way stream used to narrate progress and deliver the final complete View from the server to the browser.
          staging
          A new temporary folder where Aluna assembles and checks a proposed version before moving it to its permanent version path.
          ticket / lease
          A ticket is a build’s place in line. A lease is temporary proof that the request currently owns the database write turn.
          tombstone
          A durable cleanup record saying the capability is already deleted while some files or resources still need removal. It prevents rebuilding the same capability name until cleanup finishes.
          View
          The complete capability interface shown in Aluna’s content area. Its structure is separate from the live records loaded into it.
