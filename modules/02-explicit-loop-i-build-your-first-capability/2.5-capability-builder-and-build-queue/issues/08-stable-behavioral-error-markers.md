# Stable behavioral error markers

## What to build

Harden the behavioral tier so validation-error behavior is asserted through the
spec's structured contract, not through brittle literal product copy.

Issue 06 correctly made the behavioral rung independent from handler code: tests
are generated from the spec's `behavior` and schema, then run against generated
handlers on the scratch database. HITL found one remaining brittleness point:
when generated tests are produced independently from the same broad behavior,
they can still invent exact user-facing error text that the generated handler is
not required to use.

Concrete example from HITL:

- Generated handler rejected missing required `title` and `body` values with an
  error fragment like
  `<div class="error" data-role="error">Missing required field(s): title, body</div>`.
- Generated behavioral test expected exact copy such as `Title is required` and
  `Body is required`.
- The handler behavior was semantically right, but the test failed on copy that
  is not part of the durable contract.

The fix is to make validation-error expectations first-class in the spec and
have both downstream AI generations consume that same structured contract:

- **Spec generation** records stable behavioral error cases for validation
  behavior: action, trigger, stable error code, affected fields, and expected
  semantic markers. Product copy is not the contract.
- **Handler generation** emits the semantic markers when rejecting invalid input.
  For missing required fields, use one stable marker shape such as
  `data-role="error"`, `data-error-code="missing_required_fields"`, and
  `data-error-fields="title body"` on the returned error fragment, or an
  equivalent repo-wide marker shape documented in code.
- **Behavioral test generation** asserts those stable markers and fields from
  the spec contract. It must not assert exact user-facing error strings, and it
  still must not consume handler code.

This should preserve the point of the behavioral tier: the test and the handler
are generated independently, but they share the spec as the source of truth for
observable behavior.

      baseline skip behavior, scratch database execution, fail-closed outcome,

## Implementation notes

_2026-06-24 — implemented and verified._

- Added a strict spec-owned `behavioral_errors` contract in
  `src/registry/spec.ts`, with the repo-wide marker shape:
  `data-role="error"`, `data-error-code="missing_required_fields"`, and
  `data-error-fields="<affected fields>"`. The spec validator requires one
  missing-required-fields case covering every required field when a capability
  has required fields.
- Persisted the new contract through the registry with migration
  `0003_capability_registry_behavioral_errors`; existing stored rows with the
  default empty column are rehydrated with the standard missing-required-fields
  contract from their stored schema. The router's row-to-spec projection now
  carries `behavioral_errors` into the scoped data tool.
- Updated spec generation to require `behavioral_errors` and updated handler
  generation so `create` handlers are instructed to return semantic error
  markers while leaving product-voice copy unconstrained.
- Extended behavioral test generation and execution with validation-error
  cases. Generated validation-error tests copy `expectedError` from the spec
  contract and the runner asserts marker attributes, stable error code, and
  affected fields. The runner rejects validation-error cases that try to assert
  product-copy substrings instead.
- Preserved behavioral-tier independence: the test-generation prompt includes
  only `behavior`, `schema`, and `behavioral_errors`; handler source and
  generated fragments still never enter test generation.
- Extended fake-provider coverage for multiple required fields, copy variation
  with correct markers, missing/wrong marker failure, default-on/off behavior,
  scratch execution, and the homepage demo's developer-only
  `build-error-preview` diagnostic for marker mismatches.
- Follow-up from live provider HITL: OpenAI Responses rejected the behavioral
  test schema when `expectedError` was optional, because its structured response
  format requires every object property to appear in `required`. Behavioral test
  cases now require `expectedError` on every case and define it with
  `.nullable()`, using `null` for normal success cases. `src/builder/gate.test.ts`
  asserts the generated JSON Schema's `cases.items.required` includes every
  property, including `expectedError`, and that the field accepts `null`.

   `OMNI_BEHAVIORAL_TIER=off OMNI_API_KEY=<your key> bun run dev`, run the same
   prompt, and confirm the gate preview records `behavioral:skipped` and
   `behavioral.tier: "off"`.
